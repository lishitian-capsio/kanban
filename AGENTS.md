TypeScript principles
- No any types unless absolutely necessary.
- Check node_modules for external API type definitions instead of guessing.
- Prefer SDK-provided types, schemas, helpers, and model metadata over local redefinitions. For things like reasoning settings, use the pi adaptation layer (`src/agent-sdk/kanban/`) or the SDK's source of truth whenever possible instead of recreating unions, support checks, or shapes in Kanban.
- NEVER use inline imports. No await import("./foo.js"), no import("pkg").Type in type positions, and no dynamic imports for types. Always use standard top-level imports.
- NEVER remove or downgrade code to fix type errors from outdated dependencies. Upgrade the dependency instead.

Code quality
- Write production-quality code, not prototypes
- Break components into small, single-responsibility files. 
- Extract shared logic into hooks and utilities. 
- Prioritize maintainability and clean architecture over speed. 
- Follow DRY principles and maintain clean architecture with clear separation of concerns.
- In `web-ui`, prefer `react-use` hooks (via `@/kanban/utils/react-use`) whenever possible
- Before adding custom utility code, evaluate whether a well-maintained third-party package can reduce complexity and long-term maintenance cost.

Architecture opinions
- Avoid thin shell wrappers that only forward props or relocate JSX for a single call site.
- Prefer extracting domain logic (state, effects, async orchestration) over presentation-only pass-through layers.
- Do not optimize for line count alone. Optimize for codebase navigability and clarity.

Git guardrails
- NEVER commit unless user asks.

web-ui Stack
- Kanban web-ui uses Tailwind CSS v4 for styling, Radix UI for accessible headless primitives, and Lucide React for icons.
- Custom UI primitives live in `src/components/ui/` (button, dialog, tooltip, kbd, spinner, cn utility).
- Toast notifications use `sonner`. Import `{ toast }` from `"sonner"` or use `showAppToast` from `@/components/app-toaster`.

Styling mental model
- Use Tailwind utility classes as the primary styling system. Prefer `className` over inline `style={{}}`.
- Prefer Tailwind classes over adding custom CSS in `globals.css` when possible. Conditional Tailwind classes via `cn()` are better than CSS overrides for state-driven styling (e.g. selected/active variants). Reserve `globals.css` for things Tailwind can't express: complex selectors (sibling combinators, attribute selectors), app-level layout glue, or styles that genuinely need to cascade.
- Only use inline `style={{}}` for truly dynamic values (colors from props/variables, computed positions from drag-and-drop, runtime-dependent dimensions).
- The design system tokens are defined in `globals.css` inside `@theme { ... }`. Use Tailwind utilities that reference them: `bg-surface-0`, `text-text-primary`, `border-border`, etc.

Misc. tribal knowledge
- Kanban-owned code logs through the unified facade `src/logging/` (`createLogger(namespace, baseFields?)` → `.debug/.info/.warn/.error(message, fields?)` + `.child(fields)`); the web-ui counterpart is `web-ui/src/utils/logger.ts` (same API, console-only sink). **Do not call `console.*` for diagnostics** — the lint-free pattern is a module-level `const log = createLogger("<namespace>")`. Structured context (`taskId`/`workspaceId`/`agentId`/`error`) goes in the fields object, never string-interpolated into the message; `Error` values are unwrapped (name/message/stack), not stringified to `{}`. Levels resolve from `KANBAN_LOG_LEVEL` (default `info`; `KANBAN_DEBUG_MODE` truthy ⇒ `debug`); format from `KANBAN_LOG_FORMAT` (`pretty` on a TTY, `json` when piped). Output is **stdout/stderr by default**; set `KANBAN_LOG_FILE` truthy to also write a rotating file to `~/.kanban/logs/kanban.<date>.log` (machine-home, **never** the committed repo `<repo>/.kanban`, so no `.gitignore` rule is needed). The level/format/error-unwrap logic is pure and unit-tested (`test/logging/`, `web-ui/src/utils/logger.test.ts`); winston (already a dep) only does file rotation behind a lazy sink, so importing the module is side-effect-free. Three deliberate non-migrations: (1) **vendored omp** (`src/agent-sdk/{ai,shared,compaction,telemetry}`, `agent.ts`) keeps its OWN logger (`src/agent-sdk/shared/logger.ts` → `~/.omp/logs/`) — don't merge or touch it; (2) user-facing **CLI banner/passcode** output uses `src/cli-output.ts` `printLine` (clean stdout, no log prefix — the passcode must never hit a log file); (3) hook **wire-protocol** stdout/stderr (`src/commands/hooks.ts`, `runtime-workspace.ts`) stays raw `process.stdout.write`.
- The agent-agnostic session message model lives in `src/session/`, NOT in the pi adaptation layer. `SessionMessage` (`session-message.ts`) is a type alias of the websocket contract type `RuntimeTaskChatMessage` (`runtimeTaskChatMessageSchema` in `api-contract.ts`) — the wire schema is the source of truth, so the in-memory transcript and the `task_chat_message` broadcast payload can't drift. Streaming reconciliation (assistant/reasoning/tool fold-in + per-turn cursors) is in `session-message-buffer.ts` and operates on a generic `SessionMessageBuffer`; `SessionMessageSource` (`session-message-source.ts`) is the read/subscribe contract the runtime hub consumes. pi's `KanbanTaskSessionEntry` (`src/agent-sdk/kanban/session-state.ts`) just `extends SessionMessageBuffer` and adds the pi `summary`; `PiTaskSessionService extends SessionMessageSource`. This split is the shared base for future CLI/terminal transcripts, persistence, and resume — do NOT re-add the message model under `src/agent-sdk/kanban/`. The role enum already covers both pi's rich structure and a CLI agent's lightweight per-turn `assistant` text, so don't widen the model per-agent.
- CLI/terminal agents (claude/codex/...) now produce a real `SessionMessage` transcript too, captured in memory by `TerminalSessionManager` (which `implements SessionMessageSource`). The non-obvious part: there is no structured event stream from a PTY, so assistant text is reconstructed from the xterm headless mirror's **committed scrollback** — `TerminalStateMirror.getCommittedLines()` returns only the logical lines that have scrolled *above* `buffer.active.baseY` (the live input box/spinner in the viewport is deliberately excluded), and returns `[]` for the alternate-screen buffer. The **turn boundary** is the session-state-machine transition into `awaiting_review` (Claude's `Stop` hook → `transitionToReview` → `applySessionEvent`); each boundary folds the new committed-line delta into one `assistant` message. User messages come from the kickoff prompt (`startTaskSession`) and follow-up keystrokes in `writeInput` (committed on Enter, ANSI/backspace-stripped). All of this lives in `src/terminal/terminal-transcript-capture.ts` (pure, unit-tested) — fidelity is intentionally "good enough" (no per-tool-call parsing; the prompt echo Claude prints back may leak into the next assistant message). The hub broadcasts these over the same `task_chat_message` channel as pi via `manager.onMessage`, and `getTaskChatMessages` falls back to the terminal manager when pi has no session.
- Session transcripts are now **persisted to disk** so they survive a Kanban restart (the old "memory-only" note is obsolete). The store is `src/session/session-message-journal.ts` (`FileSessionMessageJournal`), writing append-only JSONL at `~/.kanban/workspaces/<id>/sessions/<taskId>/messages.jsonl` (one JSON `SessionMessage` per line; sibling to the `sessions.json` *summary* file). Both agents share it: it's injected into `InMemoryPiTaskSessionService` (via `CreatePiTaskSessionServiceOptions.messageJournal`, wired in `runtime-server.ts`) and `TerminalSessionManager` (constructor `messageJournal`, wired in `workspace-registry.ts`); the in-memory/test default is `NoopSessionMessageJournal`, so pi's behavior is unchanged when no journal is configured. Non-obvious bits: (1) pi re-emits the **same assistant message id** on every `message_update` token, so the journal *coalesces* a per-task `tail` and only appends on id-change / a 250ms debounce / `flush()` — naive append-per-emit would write hundreds of lines per message; (2) reads de-dupe by id (last write wins), tolerate a torn trailing line from a crash, opportunistically compact, and cap at `maxMessages` with an in-band `status` truncation marker; (3) restart history reaches the UI through the **existing** `getTaskChatMessages` path — `loadTaskSessionMessages` does `mergeSessionMessages(persisted, liveInMemory)`, so no frontend change was needed (the chat panel already fetches on open via `use-kanban-chat-session.ts`). The journal flushes on `dispose()` and `markInterruptedAndStopAll()`. The pi service can't be imported under vitest (agent-sdk touches `Bun.env` at import → existing tests mock it), so pi persistence is covered by a Bun round-trip, not a vitest test; the journal core and the terminal path are vitest-tested. Agent **session resume** (re-attaching pi/CLI to a prior conversation) is still a later task — this is view/read-back persistence only.
- The **home (sidebar) chat is multi-thread-capable at the foundation layer**. The synthetic home session id (`src/core/home-agent-session.ts`) gained an optional fourth segment: `__home_agent__:<workspaceId>:<agentId>:<threadId>`. **Backward-compat is load-bearing**: `createHomeAgentSessionId(ws, agent, threadId?)` emits the *legacy three-segment* id when `threadId` is omitted or equals `DEFAULT_HOME_THREAD_ID` (`"default"`), so the default thread's on-disk taskId is unchanged and existing transcripts/`sessions.json`/resume reconnect with no migration. `parseHomeAgentSessionId` parses positionally (workspaceId, agentId, optional threadId) — this is why `append-system-prompt.ts`'s agentId lookup uses the parser, NOT `parts.at(-1)` (which would grab the threadId on a 4-segment id). The thread **registry** is persisted per-workspace at `<repoPath>/.kanban/workspaces/<id>/threads.json` (a `{ threads: RuntimeHomeChatThread[] }` doc, sibling to `sessions.json`): pure list/create/rename/close ops live in `src/session/home-thread-registry.ts` (I/O-free, vitest-tested); `src/state/workspace-state.ts` owns persistence (`loadWorkspaceHomeThreads` / `mutateWorkspaceHomeThreads` — locked read→mutate→atomic-write, resolving `repoPath` via `resolveRepoPathForWorkspaceId` like the other per-workspace loaders post-T1); `src/session/home-thread-store.ts` (`HomeThreadStore`) is the orchestration seam with **injected** persistence + an `onCloseSession` callback (so it's unit-testable without touching `~/.kanban`). Closing a thread is a **hard close**: `TerminalSessionManager.closeTaskSession` / `InMemoryPiTaskSessionService.closeTaskSession` stop the process, drop the in-memory entry, and delete the transcript via `journal.clear(taskId)`. Wiring is in `runtime-server.ts` (`getScopedHomeThreadStore`, one store per workspace, `onCloseSession` routes cleanup to the agent that actually backs the session via `parseHomeAgentSessionId`); it's exposed through the `createRuntimeApi` deps but **not yet consumed** — tRPC endpoints + the multi-thread sidebar UI are deliberately later tasks, this layer is id-scheme + registry + close-lifecycle only.
- The web-ui runtime state stream is a **granular external store**, not a monolithic `useReducer`. `runtime/runtime-stream-store.ts` holds a module singleton driven by the SAME pure `runtimeStateStreamReducer` (the transition source of truth, behavior-identical to the old reducer), but `dispatchRuntimeStreamAction` diffs the next store **per field reference** and only wakes that field's listener set — plus a separate **per-taskId** chat listener map, so a `task_chat_message` token for task A never re-renders task B's panel or the board-sync badge. Consumers read slices via `useSyncExternalStore` selector hooks (`useRuntimeProjects`, `useRuntimeWorkspaceState`, `useTaskChatMessages(taskId)`, `useRuntimeBoardSyncStatus`, …) — the model is `stores/workspace-metadata-store.ts`. `use-runtime-state-stream.ts` is now just the connection driver `useRuntimeStreamConnection` (one socket, dispatches into the store, returns nothing). **The load-bearing rule: a high-frequency slice must be subscribed inside the LEAF component that shows it, never in a hook called at App level** — a `useSyncExternalStore` subscription re-renders the component whose fiber called it, so subscribing chat at App level (directly or via an App-level custom hook) would re-render the whole tree and defeat the split. That's why chat lives in `CardDetailView` + the `HomeSidebarAgentPanel` **component** (converted from the old `useHomeSidebarAgentPanel` hook-returning-JSX precisely so its chat/`kanbanSessionContextVersion` subscriptions sit in their own fiber), and the board-sync badge lives in `components/home-board-sync-control.tsx` (owns `useBoardSync`, mounted by `TopBar`). `App`/`use-project-navigation` subscribe ONLY to low-frequency slices (projects/workspaceState/metadata/readyForReview/error/connected/snapshot). The singleton seeds `currentProjectId` from the URL at module load (mirrors the old `useReducer` initializer) so the first render before the connection effect doesn't flash a loading state. `task_sessions_updated`/`workspace_state_updated` still flow through `workspaceState` → `useWorkspaceSync` → local `board`/`sessions` state (board ref guarded by revision), unchanged. Test: `runtime/runtime-stream-store.test.tsx` proves the per-slice isolation with render counters.
- Requirement data is **sharded by requirement id on disk**, not stored as single JSON files. Per workspace (`<repo>/.kanban/workspaces/<id>/`) the three former single files are now directories: `requirements/<reqId>.json` (one item), `requirement-versions/<reqId>.json` (that req's full `RuntimeRequirementVersion[]`), `requirement-task-links/<reqId>.json` (that req's links). This kills cross-branch git merge conflicts when different requirements are edited on different branches. The in-memory contract is UNCHANGED — `RuntimeRequirementsData`/`...VersionsData`/`...TaskLinksData` are still the aggregates; only the on-disk shape changed. Read/write live in `src/state/requirement-store.ts` (typed) on top of the generic `src/state/sharded-json-store.ts` (`readShardDir`/`writeShardDir`: a dir mirrors a `Map<id,value>`; `writeShardDir` deletes any `<id>.json` absent from the map and relies on `writeJsonFileAtomic`'s content-compare to skip unchanged files). `workspace-state.ts` reads shards when the shard dir exists, else falls back to the repo single file, else the T1 `~/.kanban` single-file fallback. Non-obvious: (1) the requirements list is rebuilt by sorting on `order` (versions/links aggregate order is cosmetic — lookups filter by id); (2) a `delete` version is appended *after* the item is gone, so **a version shard outlives its deleted requirement** — never delete version shards on requirement delete (an emptied *link* shard IS deleted); (3) one-time migration `migrateRequirementsToShards` runs in `prepareRepoRuntimeHome` *after* T1's `migrateWorkspaceDataFromLegacyHome`, under the workspace-dir lock, with a cheap "no single files present → skip the lock" fast path; it splits a repo single file into shards then deletes it, so the chain is machine-home single file → (T1 copy) → repo single file → (this) → repo shards. The `.kanban/.gitignore` is denylist-style so the shard dirs are committed by default (no rule needed). Tests: `test/runtime/{sharded-json-store,requirement-store}.test.ts` + `test/integration/requirements-sharding.integration.test.ts`.
- Kanban's native agent is `pi` (oh-my-pi, source-embedded in `src/agent-sdk/`). The old embedded Cline SDK runtime (`src/cline-sdk/` + `@clinebot/*`) and the external `cline` CLI agent option have both been removed; `runtime-api.ts` / `runtime-server.ts` and the catalog (`src/core/agent-catalog.ts`) only know `pi` plus the external CLI agents (claude/codex/droid/kiro/gemini/opencode). The pi adaptation layer (`src/agent-sdk/kanban/`) maps omp `AgentEvent` to Kanban's `RuntimeTaskSessionSummary` contract.
- The token `cline` survives in two unrelated places — never blanket-rename it: (1) the **native agent id was historically `cline`, now `pi`** (web-ui shares `RuntimeAgentId`/`RuntimeKanbanOauthProvider` from `src/core/api-contract.ts` via the `@runtime-contract` tsconfig alias, so narrowing the agent enum ripples into `web:typecheck`); (2) the **live Cline-hosted model provider/account** (`runtimeKanbanOauthProviderSchema` `"cline"`, `providerId`/`oauthProvider`, `api.cline.bot`/`app.cline.bot`/`data.cline.bot`) that the `pi` agent uses for managed models, credits, OAuth, and Featurebase — keep all of those.
- Kanban is launched from the user's shell and inherits its environment. For agent detection and task-agent startup, prefer direct PATH checks and direct process launches over spawning an interactive shell. Avoid `zsh -i`, shell fallback command discovery, or "launch shell then type command into it" on hot paths. On setups with heavy shell init like `conda` or `nvm`, doing that per task can freeze the runtime and even make new Terminal.app windows feel hung when several tasks start at once. It's fine to use an actual interactive shell for explicit shell terminals, not for normal agent session work.
- **The runtime is Bun-only; there is no Node fallback and no native dependencies.** `dist/cli.js` / the `kanban` bin ship `#!/usr/bin/env bun`; dev is `bun src/cli.ts`; the `service` daemon invokes that bin. `node-pty` (the last native-compiled dep) was removed and the `packages/desktop` Electron app — the one path that ran the runtime under Node — was deleted (see `.plan/docs/node-pty-removal-feasibility-investigation.md`). `src/terminal/pty-session.ts` (`PtySession`) now has a **single backend: Bun's native `Terminal`** (`Bun.spawn(argv, { terminal })`); `spawn` **throws** if the Bun Terminal API is absent — that's deliberate, do NOT "fix" it by re-adding node-pty or a Node fallback. Windows `.cmd`/`.bat` cmd.exe wrapping (`src/core/windows-cmd-launch.ts`) is the only launch-rewrite path. PtySession tests inject a fake `globalThis.Bun` so they pass under both `bun vitest run` (local) and `npx vitest run` (CI on Node).
- If CI hangs on Node 22 after tests seem to finish, suspect a live subprocess or SDK-host startup path before assuming a slow test body. Read `.plan/docs/node22-ci-hanging-tests-investigation.md` before repeating that investigation. The old `test/runtime/cline-sdk/` test suite was the prior culprit (now deleted) because it booted the real Cline SDK host.
- When Kanban runs on a headless remote Linux instance (for example over SSH+tunnel), native folder picker commands may be unavailable (`zenity`/`kdialog`). Treat this as a normal remote-runtime limitation and use manual path entry fallback instead of requiring desktop packages.
- Commander nested subcommands: if a child subcommand re-declares an option already declared on its parent (e.g. `--project-path` on both a `foo` command and its `foo apply` child), commander silently routes the passed value to the *parent* command, so the child action's plain `options` arg has it `undefined`. Read it via `this.optsWithGlobals()` inside the child action (use a regular `function`, not an arrow, so `this` is the Command). This first bit during the (now-removed) requirement review/reconcile `apply` subcommands and looked like a workspace-resolution bug — watch for it whenever you add a nested `<command> <subcommand>` that shares a parent option.
- The board is persisted **sharded**, not as one `board.json`. Each task is its own committed file `<repo>/.kanban/workspaces/<id>/tasks/<taskId>.json` (durable spec: title, prompt, `column`, `rank`, baseRef, auto-review, `dependsOn`); `board.json` was **repurposed into a cards-free layout manifest** (`{ version, columns: [{id,title}] }`). The non-obvious core: **the wire contract `runtimeBoardDataSchema` is unchanged** — sharding lives entirely in `src/state/task-shard-store.ts`, which *assembles* the ordered `{columns:[{cards}], dependencies[]}` board on read and *decomposes* it on write, so `rank` and the per-task `dependsOn` decomposition never reach the frontend/contract (zero web-ui change). Ordering is a fractional `rank` string (`src/state/task-rank.ts` wraps the `fractional-indexing` pkg) reconciled to keep existing ranks where monotonic, so append/move/no-op rewrite only the task that actually moved — that's what makes per-task files free of git merge conflicts. Dependencies (one-way backlog→non-backlog) are stored on the **from-task's** file as `dependsOn` (implicit `fromTaskId`), so distinct tasks' edges never share a file. Migration from the old single-file board runs eager + idempotent in `prepareRepoRuntimeHome` (after the T1 legacy-home copy) under the workspace-dir lock via `convertBoardToShards`, gated by the cheap `boardNeedsSharding`; `loadShardedBoard` also stays read-compatible with a legacy single-file `board.json` (and the machine-rooted fallback) without writing. A torn/invalid task shard is silently skipped (crash tolerance), but a malformed **old-shape** `board.json` still fails loudly with the `board.json`-labeled error (`parseBoardDataOrThrow`) the pre-shard read path had. `tasks/` is committed because the `.kanban/.gitignore` denylist doesn't match it; `sessions.json`/`meta.json`/`worktrees/`/`*.lock` stay gitignored as before.
- In-process outbound proxy is split from subprocess proxy, and the **in-process holder (`config/proxy-fetch.ts`) is the single source of truth** for the runtime's own fetch. A `globalThis.fetch` monkey-patch installed once in `startServer()` (before provider SDKs load) reads a mutable holder per request and injects the proxy engine-natively — **Bun ⇒ `{ proxy }` option, Node ⇒ `{ dispatcher: new ProxyAgent(url) }`** (undici). The holder is updated by `setRuntimeProxyStateFromConfig` at `trpc/runtime-api.ts` saveConfig + `server/workspace-registry.ts` startup, so settings changes take effect on the next request with no restart. The patch covers everything that ultimately calls `globalThis.fetch` — provider SDKs (OpenAI/Anthropic/Google/Bedrock-custom-fetch) and the ~26 bare `fetch()` OAuth/token/model-discovery sites. It does NOT cover non-fetch transports (Cursor's HTTP/2 stub; AWS SDK — not used). It passes through untouched when proxy is disabled, when the caller already set `dispatcher`/`proxy` (e.g. `getRuntimeFetch`'s CA-pinned dispatcher), and for loopback/`NO_PROXY` hosts (`shouldBypassProxy`).
  - **The non-obvious Bun trap that makes live switching/disable work:** Bun 1.3.x's global fetch DOES honor `HTTP(S)_PROXY` env and latches it for the process lifetime; `delete process.env.*` does NOT un-latch it and `{proxy:""}` per request does NOT force direct (both fall back to Bun's boot snapshot). The fix: `installProxyFetch()` calls `stripInheritedProxyEnv()` which sets the four URL keys to **`""`** (NOT delete — an empty-string assignment before the first fetch is the only thing that overrides the boot capture; NO_PROXY is preserved). And the runtime **never writes proxy URLs into its own `process.env`** — the old `applyProxyToProcessEnv` was removed because writing the env re-latched Bun and broke disable/switch. So enabled ⇒ explicit `{proxy:url}` option; disabled ⇒ genuinely direct, even when launched from a shell that exported a proxy.
  - **Subprocesses** still get proxy via explicit per-spawn env merge (never via the runtime's `process.env`, which is now clean): terminal agents use `config/proxy-env.ts` `buildProxyEnvVars` (session-manager); self-update and shortcut spawns merge `buildSubprocessProxyEnv()` (reads the same holder). The Bun-level routing proof is `scripts/proxy-live-switch-check.ts` (`npm run test:proxy-live`) — a parent/child listener test proving enable→switch→disable→NO_PROXY live, even with an inherited shell proxy; vitest can't (it mocks `globalThis.fetch`). This is the working successor to the abandoned `feat/live-proxy-switching` branch — **no always-on local proxy hop**.
- **Codex custom-provider config is split across TWO mechanisms — don't fold them.** Provider env injection (`unified-proxy/env-injector.ts` `buildAgentProviderEnv`, central in session-manager) supplies the **secret** as per-spawn env (`OPENAI_API_KEY`) + `OPENAI_BASE_URL`. Separately, the codex adapter calls `src/terminal/codex-home-projector.ts` `projectCodexHome` to write an **isolated** `CODEX_HOME/config.toml` at `<machine-home>/codex-home/<taskId>/` (stable per task so resume works; never the user's global `~/.codex`). That config.toml carries the rich native settings Codex env vars can't express — a named `[model_providers.<id>]` (`name`/`base_url`/`wire_api`/`env_key`) plus `model`/`model_reasoning_effort`/`model_reasoning_summary`/`model_supports_reasoning_summaries`/`preferred_auth_method` — and references the key only via `env_key = "OPENAI_API_KEY"`, so **no API key is ever written to disk or committed state**. `OPENAI_BASE_URL` from env-injector is dead for codex once a named non-`openai` provider is projected (it only reconfigures Codex's built-in `openai` provider); the API key env is what the env_key consumes. Defaults for third-party OpenAI-compatible providers: `wire_api = "responses"`, reasoning summaries explicitly OFF (`"none"` + `false`), `preferred_auth_method = "apikey"`. **Official login (and any agent with no custom provider) projects nothing → native `~/.codex` is preserved** — `projectCodexHome` returns `null` and the adapter sets no `CODEX_HOME`. Both `env-injector` and the projector call the shared `resolveAgentProvider`; both independently throw `IncompatibleAgentProviderError` for a non-openai protocol (the projector imports the class from env-injector).
- **Agent session resume is per-agent and NOT uniform — Codex captures its id AFTER launch, Claude pins one at launch.** Claude accepts a Kanban-minted UUID via `--session-id` (re-attach later with `--resume <id>`), so its id is known synchronously in `claudeAdapter.prepare` and returned as `PreparedAgentLaunch.agentSessionId`. **Codex's interactive TUI has no launch-time session-id pin** (verified `codex resume --help`: id is only a *positional* — `codex resume <SESSION_ID> [PROMPT]`). So Codex resume is **capture-after-launch**: `src/terminal/codex-session-capture.ts` polls `<CODEX_HOME>/sessions/<YYYY>/<MM>/<DD>/rollout-<ts>-<uuid>.jsonl` for the newest rollout whose first-line `session_meta.payload.cwd` matches the task worktree (cwd disambiguates the shared `~/.codex` default-login case — each task = unique worktree), reads `payload.id`, and the adapter exposes this as `PreparedAgentLaunch.captureAgentSessionId`. `session-manager.ts` runs it fire-and-forget after spawn (`captureAndApplyAgentSessionId`), **guarded by `active.session` identity** (a restart between launch and capture must not persist a stale id), then `updateSummary({agentSessionId})` — persisted through the SAME `listSummaries()`→`sessions.json` path Claude uses (no special persistence). Capture runs on **every** launch (fresh + resume) so it self-heals whether `codex resume` reuses the id or writes a fresh rollout. The effective sessions dir tracks the projected custom-provider `CODEX_HOME` (see the codex-home-projector note above) else inherited `$CODEX_HOME`/`~/.codex`, so it always matches where Codex actually wrote. On restart the adapter emits `resume <id>` (priority over the `--last` trash fallback).
- **Board data lives on its own git branch (`kanban/board`), decoupled from the code branch (方案 C).** Design+rationale: `.plan/docs/board-branch-decoupling.md` (tag `design/board-branch-decoupling`). The problem it fixes: the runtime constantly rewrites `tasks/*.json` (no commit, just disk), so when `.kanban` was committed on the *code* branch the working tree was perpetually dirty and `git switch` was blocked. **Three independent git lines, never merged into each other:** (1) **code main tree** — the user's code branch in the repo root; `.kanban` is gitignored there *except* the pointer; (2) **task code** — `kanban/task/<id>` branches in `<repo>/.kanban/worktrees/<taskId>/<label>` (unchanged by this work); (3) **board data** — an orphan `kanban/board` branch checked out in a dedicated worktree at `<repo>/.kanban/worktrees/__board__/<label>/` holding ONLY `.kanban` board JSON / vault / files. The nested `…/__board__/<label>/.kanban/…` is intentional (outer `.kanban` is the code tree's, ignored; inner is the board branch's worktree content).
  - **Two-root split** (the core abstraction, `src/state/workspace-state.ts` `BoardDataLocation`): `runtimeHome` = `<repo>/.kanban` (machine-local/gitignored: `worktrees/`, `sessions*`, `meta.json`, `*.lock`, `trashed-task-patches/`) vs `boardDataHome` = `<boardWorktree>/.kanban` (committed: `workspaces/<id>/{board.json,tasks/}`, `files/docs/**`, `files/`, committed-providers, threads). `getWorkspaceDirectoryPath` stays the chokepoint — only the *root* fed into it changes. `sessions/meta/locks` deliberately stay in the runtime root (NOT the board worktree) so the board branch's tree isn't polluted with ignored runtime files. The `boardDataHome` repoint is **gated on `board-ref` presence**, so a repo that hasn't migrated behaves exactly as before.
  - **`.kanban/board-ref` is the authoritative discovery pointer** (`src/state/board-ref.ts`, JSON `{version, branch}`) — the single `.kanban` file the code branch still tracks (root `.gitignore` flips to `/.kanban/*` + `!/.kanban/board-ref`). It travels with a clone → clone-即用: startup `fetch` + `git worktree add --track`. It must NOT live on the board branch itself (chicken-and-egg). The branch-name *setting* (workspace-level, in `VaultSettingsStore`/`files/settings.json`) is editable in Settings, but `board-ref` wins; changing it runs the **non-destructive rename migration** (`renameBoardBranch`: `git branch new old` → switch worktree → archive tag `kanban/board-archive/<ts>` → push new + delete old; any failure rolls back to the old branch and leaves `board-ref` untouched, so the board never goes empty).
  - **Sync** (`src/workspace/board-sync.ts` service + `board-worktree.ts` git ops): write→debounced (~5s) commit (dedicated `kanban-board` identity, like `turn-checkpoints.ts`)→push, serialized per-repo. Sharding makes most concurrent multi-machine edits merge cleanly; a genuine content conflict is **surfaced, never auto-resolved** — the runtime `merge --abort`s to keep the shard tree clean (conflict markers would corrupt the next shard read) and reports `state: "conflict"` with local data intact. Live status flows over a dedicated `board_sync_status_updated` ws broadcast (not the workspace snapshot); the top-bar `BoardSyncStatusControl` badge renders it, and on `conflict`/`error` the badge is clickable → `BoardConflictDialog` (explanation + worktree path + retry pull/push). `RuntimeBoardSyncStatus.worktreePath` exists for that dialog.
  - **The old "auto-commit `.kanban` before switching branches" patch (task 9d884) is dead and must NOT be reintroduced** — 方案 C makes the code branch stop tracking `.kanban` data, so switching code branches is clean for free. `runGitCheckoutAction` (`git-sync.ts`) is a plain `git switch`; there is no kanban-pre-commit hook on the checkout path, by design.
- **CLI positional-id commands: resolve the id INSIDE the `runCliCommand` handler, and pass `warnings` as a long-lived array reference.** The redesign (design `.plan/docs/kanban-cli-redesign.md` §3.3) promotes ids from flags to positionals (`task update <id>`) while keeping the old flag (`--task-id`) as a deprecated alias. The resolver lives in `src/commands/cli-positional-args.ts` (`resolveRequiredId`/`resolveOptionalId` → `{id, warning?}`; required throws `CliError("invalid_argument")`). Two non-obvious traps: (1) **resolution must run inside the `async () => {…}` handler passed to `runCliCommand`, not before it** — a `CliError` thrown *outside* the handler escapes to the top-level `cli.ts` catch and prints a generic `exit 1` instead of the structured `{ok:false,error:{code}}` envelope; only errors thrown *inside* the handler get classified. (2) The deprecation `warning` is only known *after* resolution (inside the handler), but `runCliCommand` reads `options.warnings` *after* the handler settles to build the envelope and emit the stderr `⚠` note — so the action must declare `const warnings: CliWarning[] = []` up front, pass that **same array** as `options.warnings`, and have the handler `warnings.push(resolved.warning)`. Passing `warnings.length ? warnings : undefined` (a snapshot) silently drops handler-pushed warnings. `KANBAN_SUPPRESS_DEPRECATION=1` silences only the stderr line (keys off the `deprecated_` code prefix); `warnings[]` is always in the JSON envelope, on success *and* failure.
