# omp 源码级原生集成方案（v4 — Bun 迁移 + 源码物理嵌入）

## 核心策略

1. **Kanban 运行时迁移到 Bun** — 让 omp 的 `Bun.*` API 调用零适配成本
2. **将 omp 三个核心包的源码物理复制到 `src/agent-sdk/`** — 完全拥有代码，可自由修改/裁剪
3. **修复内部 import 路径** — `@oh-my-pi/*` → 相对路径
4. **用 `js-tiktoken` 替代 `pi-natives` 的 `countTokens`**

### 为什么物理嵌入而非 npm import

| | npm import | **源码物理嵌入** |
|---|---|---|
| 代码掌控 | 黑盒，只能看 node_modules | **完全拥有，可直接修改** |
| 定制化 | 无法修改内部逻辑 | **可裁剪不需要的 Provider，修改 compaction 策略等** |
| 调试 | 需跳到 node_modules | **源码就在 src/ 下，IDE 直接导航** |
| 依赖管理 | npm 版本锁定 | **无外部依赖，通过脚本同步上游** |
| bundle 控制 | 可能引入不需要的代码 | **精确控制哪些文件参与编译** |

## Phase 0: Kanban 迁移到 Bun（前置条件）

### Kanban 当前 Node.js 耦合分析（极浅）

| 耦合点 | 使用量 | Bun 兼容性 |
|---|---|---|
| `process.env` / `process.cwd()` | 大量 | 完全兼容 |
| `from "node:"` import | 仅 `update.ts` (3处) | 完全兼容 |
| `require()` | 仅 `update.ts` inline 脚本 | 完全兼容 |
| `node-pty` 原生依赖 | `pty-session.ts` | 需重新编译 |
| esbuild 构建 | `build.mjs` | 改 target 即可 |
| vitest 测试 | `vitest.config.ts` | 支持 Bun |
| `tsx` 开发工具 | `dev` script | 替换为 `bun --watch` |

### 迁移步骤

**Task 0.1**: 安装 Bun
```bash
curl -fsSL https://bun.sh/install | bash
```

**Task 0.2**: 修改 package.json
```json
{
  "engines": { "bun": ">=1.3.14" },
  "scripts": {
    "dev": "NODE_ENV=development bun --watch run src/cli.ts",
    "test": "bun vitest run",
    "build": "bun run clean && bun run web:build && bun scripts/build.mjs && ..."
  }
}
```

**Task 0.3**: 修改 build.mjs
```javascript
const shared = {
  platform: "bun",        // node → bun
  target: "bun1.3",       // node20 → bun1.3
  // 移除 CJS shim banner（Bun 原生支持 ESM + require）
};
// shebang: #!/usr/bin/env bun
```

**Task 0.4**: node-pty 重编译
```bash
npm rebuild node-pty   # Bun 下重新编译原生绑定
```

**Task 0.5**: Electron desktop 独立处理
- `packages/desktop/` 的 Electron 主进程仍是 Node.js，不受影响
- 仅 CLI 主程序迁移到 Bun

### 迁移验证
- `bun vitest run` 全量通过
- `bun run build` 产物正常
- `bun dist/cli.js` 启动正常
- node-pty PTY 会话正常工作

### 1.3 Import 路径修复

源码复制后，需修复三类 import：

**A. 跨包引用 → 相对路径**
```typescript
// 修复前
import { streamSimple, type Model } from "@oh-my-pi/pi-ai";
import { sanitizeText } from "@oh-my-pi/pi-utils";
import { countTokens } from "@oh-my-pi/pi-natives";

// 修复后
import { streamSimple, type Model } from "../ai/index";
import { sanitizeText } from "../shared/sanitize-text";
import { countTokens } from "../natives-shim";
```

**B. pi-utils 中的 Bun API → Node.js 等价实现**

由于 Kanban 已迁移到 Bun，**大多数 Bun API 调用无需修改**（Bun.env、Bun.sleep、Bun.hash、Bun.spawn 等在 Bun 运行时中原生可用）。

仅以下需微调：
- `env.ts` 中的 `Bun.env` → 保持（Bun 运行时原生支持）
- `dirs.ts` 中的 `Bun.hash` → 保持（Bun 运行时原生支持）
- `procmgr.ts` / `ptree.ts` → 不复制，无需处理

**C. 不复制的 Provider 清理**
- `register-builtins.ts` 中移除被裁剪 Provider 的懒加载 import
- `index.ts` 中移除不需要的 export

### 1.4 pi-natives 替代（natives-shim.ts）

omp 核心包仅 compaction 模块用到 `countTokens`（2 个文件，4 处调用）：

```typescript
// src/agent-sdk/natives-shim.ts
import { getEncoding } from "js-tiktoken";

const enc = getEncoding("o200k_base");

export function countTokens(input: string | string[]): number {
  if (Array.isArray(input)) {
    return input.reduce((sum, text) => sum + enc.encode(text).length, 0);
  }
  return enc.encode(input).length;
}
```

## 风险和缓解

### 中风险
1. **node-pty 在 Bun 下的兼容性** — 原生绑定需重新编译
   - 缓解：node-pty 官方支持 Bun
2. **omp 源码中 `import type { Subprocess } from "bun"`** — 类型引用
   - 缓解：安装 `@types/bun` 作为 devDependency
3. **Electron desktop 包独立运行时** — Electron 主进程仍是 Node.js
   - 缓解：desktop 包独立构建，不受 CLI 影响

### 低风险
4. **omp 上游同步** — 源码物理复制后需手动同步
   - 缓解：编写 `scripts/sync-omp-core.sh` 脚本，定期从 omp 仓库拉取更新
5. **import 路径修复遗漏** — 大量文件的跨包引用
   - 缓解：TypeScript 编译检查 + 逐模块验证

## 源码同步策略

```bash
# scripts/sync-omp-core.sh
# 从 omp 仓库拉取最新核心代码到 src/agent-sdk/
OMP_REPO="${1:-/root/code/oh-my-pi}"

# 同步 agent 核心
rsync -av --delete "$OMP_REPO/packages/agent/src/" "src/agent-sdk/"

# 同步 ai 核心（排除不需要的 Provider）
rsync -av "$OMP_REPO/packages/ai/src/" "src/agent-sdk/ai/"

# 同步 utils（排除不需要的模块）
rsync -av --exclude='ptree.ts' --exclude='procmgr.ts' --exclude='glob.ts' \
  --exclude='dirs.ts' --exclude='postmortem.ts' \
  "$OMP_REPO/packages/utils/src/" "src/agent-sdk/shared/"
```

适配层（`kanban/`）独立维护，不受同步影响。
