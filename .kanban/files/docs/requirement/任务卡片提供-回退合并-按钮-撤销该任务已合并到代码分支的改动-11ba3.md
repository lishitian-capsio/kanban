## 谁受影响

使用 Kanban 编排 coding agent 的用户（本人反馈）。

## 今天的痛点

用户反馈澄清后的精确需求：

> 有些任务卡片**已经执行了合并**（把任务分支的改动并进了代码分支）。当这次合并不该发生 / agent 改坏了，我需要在**那张任务卡片上有一个按钮，一键把这次合并回退掉**。

现状：改动已经落到代码分支上，`/rewind`（只回退对话）和「回退 worktree 到检查点」（改动前）都救不了——要撤销的是**已完成的那次合并本身**，目前只能手动敲 git，没有产品化入口。

## 现状与约束（在 Kanban 架构里的落点）

- 任务在 `kanban/task/<id>` 分支执行，**合并由 Kanban 发起**（自动审查 commit/PR，或手动完成）。所以「哪张卡做过合并、合并提交是哪个」是运行时已知或可追溯的信息 → 卡片可据此显示按钮。
- 判断某任务是否已并入当前代码分支：`git merge-base --is-ancestor` / `git branch --merged`。
- 回退实现有两条路，需在实现任务里定夺（**倾向非破坏性**）：
  - `git revert -m 1 <mergeCommit>`：生成一个反向提交撤销合并，**不改写历史、可追溯**（若已 push，首选）；
  - `git reset --hard <合并前>`：真正回到合并前，**改写历史**，仅适合未 push 的本地分支。

## 成功的样子

- 对**做过合并**的任务卡片，卡片上出现「回退合并」按钮（没合并的卡片不显示）。
- 点一下即可撤销该任务对代码分支的合并，无需手动 git。
- 行为可预期、不静默丢数据：回退前状态可找回（如归档 ref / 用 revert 保留历史），并明确告知这撤销的是「代码分支上的这次合并」。

## 范围说明

本需求聚焦**撤销已完成的合并**。相邻但不同的问题：对话回退（`/rewind` 已覆盖）、合并前把 worktree 回退到检查点、看板状态回退（误删/误移列）。合并状态检测是本按钮的**前提**（卡片得先知道自己合并过），归入本需求。
