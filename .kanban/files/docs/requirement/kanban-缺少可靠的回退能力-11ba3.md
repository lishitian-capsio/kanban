## 谁受影响

使用 Kanban 编排 agent 的用户（本人反馈）。

## 今天的痛点

用户反馈："kanban 这个系统有个问题，无法回退。"

当操作出错或想撤销时，缺少一条可靠的"回退"路径把系统恢复到之前的状态。已知的一个具体缺口：

- **`/rewind` 只回退对话，不撤销副作用。** 用 `/rewind` 能把与 agent 的对话回退到之前某一轮，但 agent 在那些轮次里已经执行的 CLI 副作用不会被撤销——例如已写入看板的任务（创建/移动列/删除）、已改动的 task worktree 文件、已产生的 commit。用户会误以为"回退"了，实际上看板和磁盘状态仍停留在最新。

（用户此次是笼统反馈"无法回退"，未指明是仅指对话回退、看板任务状态回退，还是代码改动回退；上面记录的是目前已知的具体表现，待进一步澄清可细分。）

## 成功的样子

用户能明确知道每种"回退"覆盖什么、不覆盖什么；对于确实需要撤销的操作（看板任务状态、代码改动），有一条可预期、可发现的恢复路径，不再出现"以为回退了其实没回退"的落差。
