## 谁受影响

使用 Kanban 编排 coding agent 的用户（本人反馈）。

## 今天的痛点

用户反馈："kanban 这个系统有个问题，无法回退。" 澄清后核心是**代码回退**：

> 某个任务的 worktree 里 agent 把代码改坏了，我想回到**改动之前**或**某个中间检查点**，但没有可发现、可预期的一键办法。

现状与相关约束：

- 现有的 `/rewind` **只回退对话**，不撤销 agent 已经落到磁盘的代码改动——容易"以为回退了其实没回退"。
- 任务在自己的 `kanban/task/<id>` 分支、独立 worktree 里执行；Kanban 已有**逐轮检查点（turn checkpoints）**会在任务分支上留提交点。**所以底层 git 完全能回到改动前/某个检查点，缺的是把它产品化。**
- 相关坑：直接在活跃任务 worktree 里手改文件，重启会把 worktree 重置到 HEAD，改动可能丢失——更说明需要一条"官方的"回退/恢复路径，而不是让用户手动 git。

## 成功的样子

- 在任务视图里能看到该任务的**检查点/改动历史**，一键把 worktree 代码回退到**改动前**或**任一检查点**，无需手动敲 git reset/checkout。
- 回退行为可预期、可发现，并明确它覆盖的是"代码"这一层（与对话回退、看板状态回退区分开）。
- 回退不应静默丢数据：回退前的状态可找回（例如落到一个归档 ref），避免二次误操作。

## 范围说明

本需求聚焦**代码层回退**。对话回退（`/rewind` 已覆盖）与看板状态回退（误删/误移列，暂无）是相邻但不同的问题，如需可另立需求。
